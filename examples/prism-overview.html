Prism 项目全景

一份对 Prism 语义文档 DSL 的结构化理解 — 它的定位、架构、设计哲学与成长空间。

项目定位

Prism 解决的核心问题是：AI agent 在生成结构化文档时，缺乏一套统一的视觉语言和写作纪律。Markdown 太弱，手写 HTML 太自由，而 Prism 用组件 API 把信息架构的最佳实践 编码进组件本身。

Prism 不是通用 UI 组件库，而是专门为 agent 写文档设计的工具链。它的目标读者是人类，但作者是 agent。这个定位让它可以做出对通用组件库来说太强的假设 — 比如强制视觉预算、要求先结论后证据。

这个定位的独特之处在于：它不只提供「零件」，更提供「写作方法论」。principles.md 中长达 12KB 的设计原则，本质上是一套面向 agent 的风格指南。

核心工作流

Prism 的使用流程极度简洁：agent 写一个只含 Vue 模板内容的 HTML 片段，构建脚本把它包装成一个可以直接用浏览器打开的自包含文件。

Agent 写模板 构建脚本 自包含 HTML

构建脚本做了三件事：自动提取 <h1> 作为页面标题，从 <!-- lang: xx --> 注释检测语言，然后将 Vue runtime、全部组件、全部样式内联注入。输出文件可以直接通过浏览器打开，无需任何服务器或构建工具。

Vite 以 IIFE 格式打包 src/index.ts 入口，自定义的 inlineCssPlugin 在构建阶段将所有 CSS 提取并以 <style> 注入的形式前置到 JS 包中。整个产物 dist/prism.iife.js 约 336KB（gzip 后约 103KB），包含：

  • Vue 3.5 runtime（含模板编译器）
  • 45 个 Vue SFC 组件
  • highlight.js 核心 + 14 种语言语法
  • UnoCSS 生成的工具类
  • base.css + notion.css 主题 + hljs-theme.css
function inlineCssPlugin(): Plugin {
  return {
    name: 'inline-css',
    enforce: 'post',
    generateBundle(_, bundle) {
      let cssCode = ''
      const cssFiles: string[] = []
      for (const [fileName, chunk] of Object.entries(bundle)) {
        if (fileName.endsWith('.css')) {
          cssCode += (chunk as any).source
          cssFiles.push(fileName)
        }
      }
      for (const f of cssFiles) delete bundle[f]
      if (cssCode) {
        for (const chunk of Object.values(bundle)) {
          if (chunk.type === 'chunk' && chunk.isEntry) {
            const escaped = JSON.stringify(cssCode)
            chunk.code = `(function(){var s=document.createElement("style");s.textContent=${escaped};document.head.appendChild(s)})();\n` + chunk.code
            break
          }
        }
      }
    },
  }
}

技术栈

Prism 的技术选型围绕一个核心约束：输出必须是零依赖的单文件 HTML。这意味着所有运行时都要打包进 IIFE，不能依赖 CDN 或 npm install。

组件体系

45 个组件按语义功能划分为 8 个家族。每个组件都配有 .md 文档，说明 props、slots、使用示例和「何时不该用」。这个「反模式」文档是 Prism 区别于一般组件库的地方 — 它不只告诉你能做什么，更告诉你不该做什么。

组件设计上有一个关键区分：重量级组件（decision、flow、branch、compare、steps、stacked-bar 等）受视觉预算约束 — 每个 <h2> 区域最多 1.5 个；而 轻量级组件（source、ref、badge、tag、kv）不受限制。

p-decision · p-callout — 结论优先，状态可视化。 p-finding · p-evidence · p-tracks · p-compare · p-contrast · p-myth · p-analogy — 多维度判断和论证。 p-collapse · p-tabs · p-pages · p-grid · p-card · p-divider · p-checklist · p-file-list p-source · p-ref · p-code · p-copy — 不受视觉预算限制，按代码需要使用。 p-metrics · p-bars · p-stacked-bar — 数字说话时才出场。 p-flow（线性）· p-branch（分支）· p-steps（时间线）

设计哲学

Prism 的设计原则不是「参考建议」，而是嵌入组件 API 的硬约束。这些原则的核心思想可以归纳为三条。

层次优先 — 结论必须可见，证据必须存在但折叠。读者先扫描标题和决策卡片，只在需要时展开细节。<p-collapse> 不是可选装饰，是传递完整性的主要机制。 一次交付完整 — 读者不应该需要追问。每个判断有证据，每个选择有被排除的替代方案，每个风险有缓解措施。顶层可扫描，折叠层穷尽。 颜色即语义 — 绿色 = 通过/安全，红色 = 拒绝/危险，黄色 = 探索/警告，蓝色 = 信息，紫色 = 审查。永远不要装饰性地使用颜色。

这些原则产生了一些具体的写作规则：组件之间必须有散文桥接，不允许两个重量级组件直接堆叠；callout 只用于中断（全文不超过 2–3 个）；图表要「挣得」出场资格；bug 必须有完整的复现链路。

假设一份文档有 4 个 <h2> 区，预算上限为 4 × 1.5 = 6 个重量级组件。如果当前顶层可见 7 个，需要把最弱的一个降级为散文或移入 <p-collapse>（折叠内的组件不计入预算）。<p-source> 和 <p-ref> 永远豁免。

架构要点

从源码结构看，Prism 的架构决策都服务于「单文件零依赖输出」这一核心约束。

导入所有 Vue SFC，逐一 app.component() 读取目标元素的 innerHTML 作为模板，创建 Vue 应用并挂载 挂载后自动运行 createToc()、highlight.js、copy-button 支持 light / dark / auto 三档

样式系统分为三层：base.css（740 行纯结构 CSS，无颜色）、themes/notion.css（2361 行 Notion 风格主题，通过 CSS 变量控制颜色）、hljs-theme.css（语法高亮配色）。这种分离意味着新增主题只需要写一个新的主题 CSS 文件。

当前状态与成长空间

Prism 处于 v0.2.0 阶段，核心功能已经成熟，但仍有明确的扩展方向。

45 个组件覆盖了文档写作的主要场景：决策、代码引用、数据可视化、流程图、对比评估。每个组件都有完整文档和反模式指引。 从模板到自包含 HTML 的流程稳定，有校验脚本保障一致性。 目前只有 Notion 风格一套主题。README 提到了 Linear、brutalist、editorial、terminal-dark 等方向，但尚未实现。CSS 变量体系已经为多主题做好了准备。 项目中没有看到单元测试或集成测试。对于一个 45 组件的库，组件级测试（至少是快照测试）会显著提升迭代安全性。

从 README 的 roadmap 来看，计划中的组件包括 <p-glossary>、<p-changelog>、<p-spec>、<p-test-result>，以及主题创作工具包和 agent 反馈循环。

总结

Prism 是一个定位精准、设计成熟的项目。它不试图成为通用组件库，而是针对「agent 写文档」这个具体场景，把信息架构的最佳实践编码进组件 API。45 个组件、8 大家族、一套完整的设计原则、自包含的构建管线 — 这些构成了一个闭环的创作工具链。

组件体系和设计哲学已经足够支撑日常使用。下一步的重心应该在测试覆盖和主题生态上，而不是急于扩展组件数量。
