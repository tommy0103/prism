Prism 当前项目理解 v2

基于新版 Prism skill 重新整理：这次把并行观察写成 tracks，而不是 timeline。

当前项目最重要的变化不是组件数量增加，而是组件开始覆盖 agent 写作里的语义动作：判断、证据、流程、对照、类比、误解纠正、并行观察。它已经不只是更好看的 HTML 输出，而是在约束 agent 如何组织信息。

几个当前事实可以作为这份判断的锚点。

项目模型

我现在会把 Prism 理解为三层：agent-facing skill 给出写作规则，Vue 组件词汇表承载语义，构建器把 template 和 runtime 打成可携带 artifact。

SKILL.md Prism template IIFE runtime Artifact

这个模型的关键在于：agent 写的不是自由发挥的网页，而是一套有边界的文档语言。视觉是结果，语义才是协议。

import PTracks from './components/PTracks.vue'
import PTrack from './components/PTrack.vue'
import PFinding from './components/PFinding.vue'

const components: Record<string, Component> = {
  PTracks,
  PTrack,
  PFinding,
}
 
- Breathe. Max 2-3 heavy components per viewport.
- Document-level budget. The total number of heavy components should not exceed the number of h2 sections.
- Hierarchy first. Verdict visible, evidence collapsed.

并行观察

这一段是新版组件最适合表达的内容。它不是 step-by-step 进度，而是几条并行评价线：词汇表、写作约束、样式 contract、公开文档。

p-contrast、p-branch、p-term、p-myth、p-analogy 补上了「解释复杂概念」的写作动作。 p-tracks 和 p-finding 解决了之前用 p-steps 表达并行增强脉络的问题。 重组件数量不应超过 h2 section 数量，这会迫使 agent 把弱信号降级为 prose，而不是把所有内容都做成卡片。 UnoCSS 是内部 authoring layer；对外仍是 .prism-* class 和 --p-* token。这个边界没有破坏 Prism 的文档协议感。 当前源码是 43 个 Vue SFC、29 个组件文档，但 README 的主叙事仍出现 34 components。

这正是 p-tracks 相比 p-steps 的价值：读者不会被误导成「这些事情必须按顺序完成」，而会把它理解成并行维度上的项目状态。

我看到的当前缺口

剩下的问题大多不是实现问题，而是 context drift：源码、skill、README、showcase 的公开叙事还没有完全同步。

最值得优先修的是 README 口径。 技术实现已经走在前面，但 README 首页仍把项目描述成 34 个组件。这会让外部读者误判当前词汇表规模，也会让 agent 学到旧上下文。

对新版 skill 的评价

这次使用新版 skill 的明显变化是：它把「少用组件」写进了规则，并提供了 p-tracks 这种更精确的语义逃生口。之前我会自然拿 p-steps 表达一组观察；现在应该先问这些观察是不是顺序关系。

我的结论是：Prism 下一步不一定要继续加很多视觉组件，更应该把这些语义边界固化为 context update workflow，让每次组件迭代都自动同步 README、SKILL、showcase 和 principles。

使用 p-tracks 表达并行观察，而不是复用 p-steps。 把源码证据放进 collapse，避免重组件过多。 同步 README 的组件数量和家族表。 把 component-iteration-flow.md 接入 `SKILL.md`，作为组件完成后的 context update workflow。
