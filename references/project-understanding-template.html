Prism 当前项目理解

一份面向项目作者的结构化观察：Prism 现在不只是把内容变好看，而是在为 agent 写作建立一套可组合的文档语法。

我理解当前项目的关键变化是：它不再只是给 Markdown 结果套皮，而是让 agent 先用 decision、source、flow、steps、contrast、myth、analogy 这些语义块组织思考，再输出一个可阅读、可扫描、可引用的单文件 HTML artifact。

先看几个稳定事实。它们说明项目已经具备了「可发布的小型 runtime」和「持续扩展的写作词汇表」两种特征。

项目身份

我会把 Prism 描述成一个面向 agent 的文档 UI 协议：作者写的是有语义的 HTML/Vue template，读者看到的是有层次、有证据、有交互的单文件文档。

Agent 意图 Prism DSL HTML artifact 读者理解

这个定位和普通 md2html 的差异，不在视觉细节，而在写作模型。

README 里的两根支柱已经很明确：一根是视觉层，减少 agent 临场写 CSS；另一根是写作层，让组件本身成为组织信息的 rubric。现在新增的 explainer 组件把第二根支柱补得更实了。

Prism ships production-ready components with a coherent theme.
Each component encodes a writing convention.
The DSL is the rubric.

架构理解

当前架构很干净：源码侧是 Vue 组件和 CSS 主题，使用侧是 template，构建侧把 runtime 和样式内联成单 HTML。这里的好处是 artifact 不需要服务器，也不需要目标环境安装依赖。

SKILL.md template.html build.js dist/prism.iife.js index.html

实现上，build.js 负责把模板包进 HTML；src/index.ts 负责注册所有 Prism 组件并挂载；vite.config.ts 负责把 CSS 注入 IIFE runtime。

语义词汇表

我把现有组件分成三层：基础阅读组件、工程证据组件、解释型组件。新增的解释型组件让 Prism 很明显区别于普通报告模板。

处理文档的基础节奏：折叠、分页、标签页、卡片、分割线、网格、目录。

structure scan

处理工程文档最需要的证据：源码、行号、复制、指标、流程、文件影响、测试覆盖。

source trace

处理复杂概念的理解：术语、误解纠正、类比、分支决策、中性对照。

mental model teaching

这组解释型组件是当前版本里最有战略意义的增强。它们让 artifact 不只是「漂亮的报告」，而是更像一个会引导读者理解复杂系统的界面。

CSS 与主题策略

我对当前 CSS 方向的理解是：对外保持普通 CSS contract，对内允许开发者用 UnoCSS 提高主题和样式生产力。这和我们刚才讨论的路线一致。

uno.config.ts 已经把 --p-* token 映射成 UnoCSS theme tokens。这样开发者可以用原子化工具构建 CSS，但最终用户仍然面对稳定的 .prism-* class 和 CSS variables。

 
export default defineConfig({
  plugins: [
    UnoCSS(),
    vue(),
    inlineCssPlugin(),
  ],
})
 
theme: {
  colors: {
    p: {
      bg: 'var(--p-bg)',
      surface: 'var(--p-surface)',
      text: 'var(--p-text)',
      accent: 'var(--p-accent)',
      success: 'var(--p-success)',
      warning: 'var(--p-warning)',
      danger: 'var(--p-danger)',
    },
  },
}

增强脉络

从最近提交和当前未提交改动看，项目增强集中在四条线：品牌表达、交互组件、解释型组件、主题/构建层。它不是一次性加功能，而是在补全「agent 写作系统」的各个面。

当前未提交改动也说明你正在做一件正确的事：把旧的 status timeline 示例迁移到 progress + flag，让进度和风险成为两个独立维度。

我看到的风险

这些不是方向性问题，更像发布前需要打磨的接口和文档一致性。它们影响的是使用者信任，而不是项目根基。

当前 src/index.ts 注册了 40 个 Vue 组件，但 README 仍多处写 34。这个数字会让读者怀疑文档是否滞后，建议发布前统一口径。

status="completed" 仍在 README 示例中出现。代码保留兼容很好，但主文档应该推广 progress + flag，否则 agent 会继续学旧写法。

项目已经有 --p-* token、.prism-* class、UnoCSS authoring layer，但还缺一份明确说明：哪些 selector 稳定，哪些只是内部实现。

下一步建议

我会优先把「公开叙事」和「稳定 contract」补齐。组件继续加没问题，但现在最值钱的是让外部用户相信这套 DSL 稳定、可学、可扩展。

继续保留单文件 HTML 输出，不引入运行时服务依赖。 继续把 UnoCSS 放在内部 authoring layer，而不是文档作者 API。 更新 README 的组件数量、组件家族和主示例。 给 progress + flag 写一段 migration note，说明 status 是兼容别名。 新增 CSS contract 文档：stable class、token、theme authoring boundary。 把 explainer 组件加入 README 的家族表，而不是只出现在 SKILL.md。

我的一句话理解：Prism 最有价值的地方不是「HTML 比 Markdown 好看」，而是「agent 可以用更精确的文档语法表达判断、证据和解释」。后续所有技术选择，包括 UnoCSS、主题、组件扩展，都应该服务这个语法感。
